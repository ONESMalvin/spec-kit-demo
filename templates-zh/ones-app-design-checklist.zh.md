# ONES App 设计检查清单

**用途**: 在设计 ONES App 功能时使用此清单确保遵循正确的设计模式

## 核心设计原则检查

### 用户交互模式
- [ ] 用户操作完全在 ONES 系统中（原生界面或者通过功能扩展实现的界面）
- [ ] 使用ONES App 功能扩展添加新的页面提供给用户使用
- [ ] 在ONES原生界面操作触发的功能主要通过监听ONES事件并响应实现（而非自定义 API）

### 技术集成模式
- [ ] 主要通过事件响应实现功能（而非自定义 API）
- [ ] 使用 ONES OpenAPI 进行数据操作
- [ ] 优先使用 ONES 扩展点（如需要）
- [ ] 仅存储必要的审计或配置数据（不重复 ONES 数据）

### 数据流设计
- [ ] 主要数据流1：用户操作 → ONES 事件 → App 处理 → ONES OpenAPI 调用
- [ ] 主要数据流2：用户操作 → App 页面 → ONES OpenAPI 调用
- [ ] 配置型数据流：用户操作 → App 页面 → App 自定义 API 调用 → App 存储
- [ ] 连接其他系统：用户操作 → App 页面 → App 自定义 API 调用 → 其他系统 API 调用
- [ ] 连接其他系统：其他系统 → App 自定义 API 调用 → ONES OpenAPI 调用

## 设计决策检查

### 避免的设计模式
- [ ] 重复存储 ONES 数据(非必要情况)
- [ ] 设计复杂的自定义数据模型
- [ ] 要求用户跳转到其他系统界面
- [ ] 创建与 ONES 功能重复的界面

### 推荐的设计模式
- [ ] 事件驱动的自动化处理
- [ ] 基于 ONES 现有数据的计算和聚合
- [ ] 通过 ONES 扩展点提供配置（如需要）
- [ ] 透明地增强 ONES 现有功能

## 具体检查项

### 功能设计
- [ ] 功能是否可以通过监听 ONES 事件实现？
- [ ] 是否可以通过调用 ONES OpenAPI 完成数据更新？
- [ ] 用户是否可以在 ONES 界面中完成所有操作？
- [ ] 是否需要创建自定义的用户界面？

### 数据设计
- [ ] 是否创建了与 ONES 数据重复的存储？
- [ ] 是否只存储必要的审计或配置数据？

## 常见错误模式

### 拒绝微服务思维
- 创建独立的 REST API
- 设计复杂的数据模型
- 要求用户调用自定义接口

### 拒绝替代思维
- 创建与 ONES 功能重复的界面
- 要求用户离开 ONES 系统
- 设计独立的用户流程

### 采用插件思维
- 响应 ONES 事件
- 增强 ONES 现有功能
- 保持用户操作的一致性

## 使用指南

1. **设计前检查**: 在开始设计前，先回答核心设计原则检查
2. **设计后验证**: 完成设计后，使用具体检查项验证
3. **避免错误模式**: 确保没有采用微服务或替代思维
4. **坚持插件思维**: 始终以增强 ONES 功能为目标

## 示例

### 好的设计
- 监听 `ones:project:issue:updated` 事件
- 自动计算父任务故事点总计
- 调用 ONES OpenAPI 更新父任务
- 用户只需在 ONES 中编辑故事点

### 不好的设计
- 创建 `/api/v1/story-points/update` 接口
- 要求用户调用自定义 API
- 创建复杂的数据模型
- 用户需要学习新的操作方式
